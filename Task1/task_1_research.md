# Задание 1. Исследование моделей и инфраструктуры

Дата проверки источников: 2026-04-11.

## 1. Контекст кейса

### Кто заказывает решение

- Основной заказчик: CTO / Head of Platform / владелец внутренней базы знаний.
- Бизнес-спонсоры: engineering management, support lead, GRC / audit function.

### Для кого делается решение

- Разработчики: архитектурные решения, API, описание сервисов.
- Поддержка: типовые инциденты, runbook, known issues.
- Менеджеры и аналитики: регламенты, правила, процессы.
- Новые сотрудники: onboarding.

### Ограничения

1. Приватность. Часть документов точно не должна уезжать во внешний API без явной классификации.
2. Стоимость. Пилот не тянет отдельный GPU-контур.
3. Сроки. Это учебный спринт, важнее добраться до рабочего бота, чем собрать enterprise-архитектуру.
4. Рост базы. После пилота не хочется переписывать стек целиком.
5. Управляемость. Нужно видеть, что проиндексировано и как обновляется.

## 2. Критерии оценки

| Критерий | Почему важен здесь |
| --- | --- |
| Качество ответа | Бот должен отвечать по базе, а не галлюцинировать. |
| Задержка | Сценарий похож на внутренний поиск, долгие ответы ломают UX. |
| Стоимость владения | Токены vs собственная инфра — надо явно сравнить. |
| Простота развёртывания | За спринт без тяжёлого DevOps и GPU-контура. |
| Риски по данным | Часть документов потенциально чувствительна. |
| Vendor lock-in | Миграция между моделями должна быть недорогой. |
| Пригодность для учебного проекта | Решение нужно не обосновать, а довести до рабочего. |

## 3. Сравнение LLM

### Варианты

- Локально: `qwen2.5:7b-instruct` через `Ollama`. Этот runtime реально используется в Task 4.
- OpenAI: `gpt-5-mini` как актуальный low-latency / cost-sensitive SKU.
- Yandex: `YandexGPT Lite` / `YandexGPT Pro` в Yandex AI Studio.

### Сводная таблица

| Вариант | Качество ответа | Скорость | Стоимость | Развёртывание | Вывод для кейса |
| --- | --- | --- | --- | --- | --- |
| Локально, `qwen2.5:7b-instruct` | При жёсткой привязке к контексту уровень рабочий, но managed API в среднем заметнее. | На CPU медленно. На Apple Silicon через `Ollama` хватает для REPL. | Нет платы за токены, есть плата за железо и сопровождение runtime. | Сильно проще полного self-hosted стека: всё поднимается через уже установленный `Ollama`. | Реальный вариант для локального учебного сценария и фактической реализации Task 4. |
| OpenAI, `gpt-5-mini` | Практичный managed-вариант по балансу качества, скорости и цены. OpenAI позиционирует его под low-latency / high-volume. | Высокая, managed API, локальный GPU не нужен. | На 2026-04-11: $0.25 за 1M input, $2.00 за 1M output. Для пилота дёшево, но плата постоянная. | Проще всего: HTTP API, SDK, минимум инфраструктуры. | Базовый выбор для быстрого пилота, если отправлять контекст в облако допустимо. |
| Yandex Cloud, `YandexGPT Lite` / `Pro` | Разумно при большой доле русскоязычных документов. Контекст common instance — 32K. | В common instance latency плавает: ресурсы shared. | На 2026-04-11: `Lite` в sync — $0.001639344 за 1K токенов, `Pro 5.1` — $0.006557376 за 1K (международный прайс, без VAT). Токенизация под русский, «токен к токену» сравнивать с OpenAI некорректно. | Проще локального, но добавляет ещё один облачный периметр поверх AWS. | Альтернатива, если нужен именно Yandex Cloud или корпус почти весь на русском. |

### Что беру в итоге

Managed cloud LLM: в учебном пилоте самый тяжёлый кусок инфраструктуры — это как раз локальный инференс, и его проще отдать наружу. Базовый выбор — `gpt-5-mini`: цена низкая, latency предсказуемая, API без сюрпризов. Если eval покажет нехватку качества, модель меняется без миграции индекса.

Локальный стек оставляю как план Б: он нужен только при жёстком запрете на outbound или когда требуется полностью local-only demo. YandexGPT работоспособен, но по кейсу QuantumForge ничего не выигрывает у варианта C — это второй облачный периметр поверх AWS.

### Оговорка

Сравнение качества без собственного eval-набора — это не benchmark. Выводы — инженерная оценка по официальным спецификациям, ценам и требованиям кейса.

## 4. Сравнение моделей эмбеддингов

### Варианты

- Локально: `sentence-transformers/all-MiniLM-L6-v2`.
- Облачно: `text-embedding-3-small` от OpenAI.

### Сводная таблица

| Вариант | Индексация | Качество | Стоимость | Вывод |
| --- | --- | --- | --- | --- |
| `all-MiniLM-L6-v2` | Для локального пилота end-to-end быстрее: нет сетевых round-trip. Модель 384-мерная, заточена под semantic search. | Хватает для учебного RAG и пилота. Важное ограничение: модель рассчитана на предложения и короткие абзацы, вход длиннее 256 word pieces обрезается. | Бесплатно по токенам, только CPU и RAM. | Самый удобный путь: дёшево, локально, данные не улетают. |
| `text-embedding-3-small` | API с batch-режимом, но появляется сетевая задержка и зависимость от сервиса. | По OpenAI сильнее, чем `ada-002`: MIRACL 31.4 → 44.0, MTEB 61.0 → 62.3. По умолчанию 1536, можно укоротить через `dimensions`. | На 2026-04-11: $0.02 за 1M токенов. Недорого, но счёт постоянный и документы улетают во внешний API. | Смысл есть, если retrieval станет узким местом. Для спринта и synthetic KB — избыточно. |

### Что беру в итоге

`sentence-transformers/all-MiniLM-L6-v2`. Модель маленькая, крутится локально, платы за токены нет, на базе из 30+ документов точности достаточно. `text-embedding-3-small` — план Б: если на реальных запросах локальный retrieval промахивается, переезд простой, индекс в любом случае пересобирается.

## 5. Сравнение FAISS и ChromaDB

| Вариант | Скорость | Внедрение и поддержка | Удобство | Стоимость | Вывод |
| --- | --- | --- | --- | --- | --- |
| FAISS | Similarity search — сильная сторона: несколько индексов с балансом скорости, качества и памяти. Есть CPU и GPU. | Для single-node — самый простой вариант. Но это библиотека индекса, не knowledge-store: метаданные и персистентность — соседние задачи. | Удобен при простой логике: локальные файлы, локальный индекс, top-k поиск. | Самый дешёвый: без сервиса, без сетевой БД, без managed-стоимости. | Для этого проекта — прямой выбор. |
| ChromaDB | Knowledge-store: документы и метаданные, фильтры, hybrid search. За это платит более тяжёлый стек. | Сложнее FAISS: больше слоёв абстракции, выше операционная поверхность, особенно в self-hosted или managed. | Если нужны коллекции, фильтры и богатый retrieval — работать с ней приятнее. | Инфра и поддержка дороже, чем у FAISS. | Для более зрелой системы нормальный кандидат, но на single-node пока избыточен. |

### Что беру в итоге

`FAISS`. Проект single-node, отдельный retrieval-сервис не нужен, индекс живёт в одном процессе рядом с пайплайном. Chroma сильнее по фильтрам и hybrid search, но эти возможности окупятся, когда появятся несколько коллекций или сложные retrieval-сценарии. Сейчас тащить их в пилот незачем.

## 6. Архитектурные варианты

### Вариант A. Полностью локальный, CPU-only

- LLM: `qwen2.5:7b-instruct`
- Embeddings: `all-MiniLM-L6-v2`
- Vector DB: `FAISS`
- Сервер: 16 vCPU / 64 GB RAM / 200 GB NVMe / без GPU

Плюсы: документы никуда не уходят, по токенам платить не надо, можно работать даже без интернета.

Минусы: на CPU 7B-модель отвечает медленно, особенно на длинных ответах. Вся эксплуатация — модель, индекс, пайплайн — на команде.

Оценка: сценарий на случай, когда облако запрещено в принципе. Для учебного спринта неудобен — основной риск в том, что до рабочего прототипа можно и не дойти.

### Вариант B. Полностью локальный, с GPU

- LLM: `qwen2.5:7b-instruct`
- Embeddings: `all-MiniLM-L6-v2`
- Vector DB: `FAISS`
- Сервер: 16 vCPU / 64 GB RAM / 1 × 24 GB VRAM / 200 GB NVMe

Плюсы: данные не уходят, с GPU 7B отвечает в секундах, нет платы за токены и зависимости от внешнего API.

Минусы: нужен GPU-сервер — железо, драйверы, runtime, обновления. Это отдельный трек эксплуатации, которого в проекте пока нет. Для учебного спринта дороже, чем оправдано масштабом задачи.

Оценка: лучший self-hosted путь для этого репозитория и мой план Б, если cloud LLM окажется недопустим. Для Task 4 удобен тем, что уже поднимается через `Ollama`.

### Вариант C. Гибрид: cloud LLM + локальный retrieval

- LLM: `gpt-5-mini`
- Embeddings: `all-MiniLM-L6-v2`
- Vector DB: `FAISS`
- Сервер: 8 vCPU / 16 GB RAM / 100 GB NVMe / без GPU

Плюсы: генерацию забирает managed API, GPU не нужен. Индекс и эмбеддинги живут у нас, поэтому чувствительные документы можно отфильтровать ещё до вызова LLM. Путь от текстов до рабочего прототипа короткий — в учебном спринте это главное.

Минусы: retrieval-контекст, который собирается под конкретный запрос, всё равно улетает в OpenAI. Появляются ключи, биллинг и отдельный разговор о том, какие документы вообще допустимо отдавать.

Оценка: мой основной вариант. По приватности не идеален, но единственный, где за спринт получится доехать до конца без ручных подвигов.

### Вариант D. Гибрид с Yandex Cloud

- LLM: `YandexGPT Lite` или `YandexGPT Pro`
- Embeddings: `all-MiniLM-L6-v2`
- Vector DB: `FAISS`; при росте требований к фильтрам — `ChromaDB`
- Сервер: 8–16 vCPU / 16–32 GB RAM / 100 GB NVMe / без GPU

Плюсы: managed-генерация без локального GPU, как и в C. Токенизация под русский: на русскоязычном корпусе расход токенов будет заметно ниже OpenAI. Есть выбор между common instance, batch и dedicated под нагрузку.

Минусы: второй облачный периметр поверх уже существующего AWS. Common instance делит ресурсы между клиентами, latency плавает. Контекст уже, чем у `gpt-5-mini`: 32K против заметно большего окна у OpenAI.

Оценка: рабочий, но у варианта C ничего не выигрывает. Имеет смысл, если проект уже живёт в Yandex Cloud или база почти вся на русском.

## 7. Итоговая рекомендация

### Стек для реализации

- LLM: `gpt-5-mini`
- Embeddings: `sentence-transformers/all-MiniLM-L6-v2`
- Векторная БД: `FAISS`
- Сервер для пилота: 8 vCPU / 16 GB RAM / 100 GB NVMe / без GPU

### Почему так

Инференс LLM — в managed API, чтобы не поднимать GPU-сервер ради учебного пилота. Retrieval и эмбеддинги остаются локально: индекс под контролем, за эмбеддинги не платим, чувствительные документы при необходимости просто не попадают в outbound-запрос. FAISS хватает с запасом на single-node и 30+ документах; фильтры и hybrid search Chroma пока некому загрузить.

### Что рассмотрено и отложено

- Локальный LLM. На 30+ документах GPU-контур не окупается. План Б на случай запрета cloud LLM.
- Облачные эмбеддинги. На таком объёме качество не окупает постоянный счёт и вынос документов в API на этапе индексации.
- ChromaDB. Сильнее по фильтрам и метаданным, но на single-node пайплайне эти возможности сейчас пустые.
- YandexGPT. Рабочий, особенно на русском, но добавляет второй облачный периметр поверх AWS без явного выигрыша.

### Правило по конфиденциальности

Для synthetic KB из Задания 2 ограничений нет: гибридный стек можно использовать как есть. Для реальной корпоративной документации сначала нужна классификация источников: `public / low sensitivity` отправляется в cloud LLM, `confidential / regulated` остаётся в local-only контуре.
