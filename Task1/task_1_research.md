# Задание 1. Исследование моделей и инфраструктуры

Дата проверки источников: 2026-04-11.

## 1. Контекст кейса

### Кто заказывает решение

- Основной заказчик: CTO / Head of Platform / владелец внутренней базы знаний.
- Бизнес-спонсоры: engineering management, support lead, GRC / audit function.

### Для кого делается решение

- Разработчики: архитектурные решения, API, описание сервисов.
- Поддержка: типовые инциденты, runbook, known issues.
- Менеджеры и аналитики: регламенты, правила, процессы.
- Новые сотрудники: onboarding.

### Ограничения, которые влияют на выбор стека

1. Конфиденциальность: часть материалов не должна покидать периметр без явной классификации.
2. Стоимость владения: пилот не должен упираться в дорогую GPU-инфраструктуру и сложную эксплуатацию.
3. Скорость запуска: проект учебный — важнее дойти до работающего результата, чем до идеальной enterprise-схемы.
4. Поддержка и масштабирование: база знаний будет расти, значит нужен стек, который не придётся полностью менять после пилота.
5. Аудит и управляемость: важно видеть, какие документы проиндексированы, как обновляется индекс и где границы доступа.

## 2. Критерии оценки

| Критерий | Почему важен именно здесь |
| --- | --- |
| Качество ответа | Бот должен отвечать по корпоративной базе, а не галлюцинировать. |
| Задержка | Пользовательский сценарий похож на внутренний поиск, долгие ответы снижают ценность. |
| Стоимость владения | Нужно сравнить оплату по токенам с расходами на собственную инфраструктуру. |
| Простота развёртывания | Для спринта важна быстрая реализация без тяжёлого DevOps и отдельного GPU-контура. |
| Риски по данным | Часть документов потенциально чувствительна. |
| Vendor lock-in | Решение должно оставлять путь к миграции, если требования изменятся. |
| Пригодность для учебного проекта | Решение нужно не только обосновать, но и успеть довести до рабочего состояния. |

## 3. Сравнение LLM

### Сравниваемые варианты

- Локальная open-weights модель: `qwen2.5:7b-instruct` как фактически выбранный self-hosted runtime для локального сценария через `Ollama`
- Облачная OpenAI-модель: `gpt-5-mini` как актуальный low-latency / cost-sensitive вариант в OpenAI API.
- Облачная Yandex-модель: `YandexGPT Lite` / `YandexGPT Pro` в Yandex AI Studio.

### Сводная таблица

| Вариант | Ожидаемое качество ответа | Ожидаемая скорость | Стоимость владения и использования | Удобство развёртывания | Вывод для кейса |
| --- | --- | --- | --- | --- | --- |
| Локально, `qwen2.5:7b-instruct` | Рабочий вариант для RAG при жёсткой привязке ответа к документам; по качеству обычно уступает managed API, но уже доступен как runnable local tag в текущем окружении через `Ollama`. | На CPU будет заметно медленнее облачных API. На Apple Silicon и локальном `Ollama` скорость приемлема для учебного REPL-сценария. | Нет оплаты за токены, но появляются расходы на локальные вычисления и сопровождение runtime. | Сильно проще полноценного self-hosted стека, потому что модель запускается через уже установленный `Ollama`. | Подходит для локального учебного сценария и фактической реализации Task 4. |
| OpenAI, `gpt-5-mini` | Практичный облачный вариант по сочетанию качества, скорости и цены. OpenAI позиционирует модель для low latency и high volume workloads. | Высокая скорость, managed API, локальный GPU не нужен. | Цена на 2026-04-11: $0.25 за 1M input tokens и $2.00 за 1M output tokens. Для пилота это недорого, но плата будет постоянной. | Запуск проще всего: HTTP API, SDK и минимум инфраструктуры. | Для быстрого пилота и учебного проекта это базовый выбор, если можно отправлять контекст во внешний API. |
| Yandex Cloud, `YandexGPT Lite` / `YandexGPT Pro` | Разумный выбор при большой доле русскоязычных документов; контекст common instance — 32K. | В common instance задержка может плавать под нагрузкой, потому что ресурсы shared. | На 2026-04-11: `YandexGPT Lite` в synchronous mode стоит $0.001639344 за 1,000 input/output токенов, `YandexGPT Pro 5.1` — $0.006557376 за 1,000 input/output токенов (международный прайс, без VAT). Токенизация подстроена под русский, поэтому сравнение с OpenAI "токен к токену" не совсем прямое. | Проще локального сценария, но в этом кейсе добавляет ещё один облачный контур поверх AWS. | Альтернатива, если нужен именно Yandex Cloud или приоритет — русский язык. |

### Вывод по LLM

- Для учебного пилота беру managed cloud LLM: она снимает с проекта локальный инференс — самую тяжёлую часть инфраструктуры.
- Базовый выбор — `gpt-5-mini`. Цена низкая, задержка предсказуемая, API без сюрпризов.
- Если позже eval-тесты покажут нехватку качества, внутри того же стека можно без миграции переехать на `gpt-5.4-mini`.
- Локальный стек имеет смысл только при жёстком запрете отдавать документы наружу или когда нужен полностью local-only demo path.
- YandexGPT работоспособен, но для QuantumForge это второй облачный периметр поверх AWS без явного выигрыша по кейсу.

### Ограничение сравнения

- Сравнение качества между OpenAI и Yandex без собственного eval-набора нельзя считать строгим benchmark.
- Выводы — инженерная оценка по официальным спецификациям, ценам и требованиям кейса.

## 4. Сравнение моделей эмбеддингов

### Сравниваемые варианты

- Локально: `sentence-transformers/all-MiniLM-L6-v2`.
- Облачно: `text-embedding-3-small` от OpenAI.

### Сводная таблица

| Вариант | Скорость создания индекса | Качество поиска | Стоимость владения и использования | Вывод |
| --- | --- | --- | --- | --- |
| `all-MiniLM-L6-v2` | Для локального пилота индексация будет быстрее end-to-end, потому что нет сетевых round-trip. Модель строит 384-мерные векторы и хорошо подходит для semantic search. | Качества достаточно для учебного RAG-проекта и пилота. Ограничение: модель рассчитана на предложения и короткие абзацы, а вход длиннее 256 word pieces обрезается. | Бесплатно по токенам, расходы только на CPU и RAM своей машины. | Для проекта это самый удобный путь: дёшево, локально и без отправки данных наружу. |
| `text-embedding-3-small` | Через API использовать удобно, есть batch-режим, но появляется сетевая задержка и зависимость от внешнего сервиса. | По официальным данным OpenAI модель сильнее, чем `text-embedding-ada-002`: MIRACL вырос с 31.4% до 44.0%, MTEB — с 61.0% до 62.3%. По умолчанию вектор имеет длину 1536, размер можно сокращать параметром `dimensions`. | Цена на 2026-04-11: $0.02 за 1M tokens. Это недорого, но всё равно появляется постоянная стоимость и вынос данных наружу. | Имеет смысл, если retrieval-качество станет узким местом. Для спринта и synthetic KB — избыточно. |

### Вывод по эмбеддингам

- Беру `sentence-transformers/all-MiniLM-L6-v2`. Модель маленькая, крутится локально, оплаты по токенам нет вообще, а для базы из 30+ документов её точности хватает.
- `text-embedding-3-small` держу как запасной путь. Если на реальных запросах локальный retrieval начнёт промахиваться, переехать на него недолго — индекс в любом случае пересобирается целиком.

## 5. Сравнение FAISS и ChromaDB

### Сводная таблица

| Вариант | Скорость поиска и индексации | Сложность внедрения и поддержки | Удобство в работе | Стоимость владения | Вывод |
| --- | --- | --- | --- | --- | --- |
| FAISS | Сильная сторона FAISS — similarity search: разные типы индексов с настройкой баланса скорости, качества и памяти. Есть CPU- и GPU-варианты. | Для single-node проекта это самый простой вариант. Минус в том, что FAISS — это библиотека индекса, а не готовое knowledge-store решение: метаданные и персистентность нужно хранить рядом. | Удобен, когда логика проекта простая: локальные файлы, локальный индекс и поиск по top-k. | Самый дешёвый путь: без отдельного сервиса, без сетевой БД и без managed-стоимости. | Для этого проекта — прямой выбор. |
| ChromaDB | Chroma — это knowledge-store: хранит документы и метаданные, поддерживает фильтры и hybrid search. За это приходится платить более тяжёлым стеком. | Сложнее FAISS: больше уровней абстракции и выше операционная поверхность, особенно в self-hosted или cloud-сценарии. | Если нужны коллекции, фильтры и более богатый retrieval, работать с Chroma удобнее. | Дороже по инфраструктуре и поддержке, чем FAISS, потому что это уже не просто библиотека индекса. | Для более зрелой системы это нормальный кандидат, но для учебного single-node проекта он сейчас избыточен. |

### Вывод по векторной БД

- Беру `FAISS`. Проект single-node, отдельного retrieval-сервиса поднимать не нужно, индекс живёт в одном процессе рядом с пайплайном.
- Chroma сильнее по фильтрам, метаданным и hybrid search, но эти возможности окупятся только когда появится несколько коллекций или более сложные сценарии retrieval. Пока этого нет — тащить их в пилот не за чем.

## 6. Рекомендуемые архитектурные варианты

### Вариант A. Полностью локальный, CPU-only

**Стек**

- LLM: `qwen2.5:7b-instruct`.
- Embeddings: `all-MiniLM-L6-v2`.
- Vector DB: `FAISS`.

**Рекомендуемый сервер**

- 16 vCPU
- 64 GB RAM
- GPU: нет
- NVMe SSD: 200 GB

**Плюсы**

- Документы никуда не уходят из контура.
- По токенам никто не платит.
- Можно работать вообще без интернета.

**Минусы**

- На CPU без GPU 7B-модель отвечает медленно, особенно на длинных ответах.
- Вся операционная нагрузка — модель, индекс, пайплайн — ложится на команду.

**Оценка**

Сценарий на случай, когда облако в принципе запрещено. Для учебного спринта неудобен: слишком много ручной работы, и главный риск — не дойти до рабочего прототипа.

### Вариант B. Полностью локальный, с GPU

**Стек**

- LLM: `qwen2.5:7b-instruct`.
- Embeddings: `all-MiniLM-L6-v2`.
- Vector DB: `FAISS`.

**Рекомендуемый сервер**

- 16 vCPU
- 64 GB RAM
- GPU: 1 x 24 GB VRAM
- NVMe SSD: 200 GB

**Плюсы**

- Данные никуда не уходят из контура.
- С GPU 7B-модель отвечает в секундах, а не минутах.
- Нет платы по токенам и нет зависимости от внешнего API.

**Минусы**

- Нужен GPU-сервер: железо, драйверы, runtime, обновления — это отдельный трек эксплуатации, которого в проекте пока нет.
- Для учебного спринта это дороже и сложнее, чем оправдано масштабом задачи.

**Оценка**

Лучший self-hosted путь для текущего репозитория и мой запасной план на случай запрета cloud LLM. Для фактической реализации Task 4 он удобен тем, что уже поднимается через `Ollama`.

### Вариант C. Гибридный, cloud LLM + локальный retrieval

**Стек**

- LLM: `gpt-5-mini`.
- Embeddings: `all-MiniLM-L6-v2`.
- Vector DB: `FAISS`.

**Рекомендуемый сервер**

- 8 vCPU
- 16 GB RAM
- GPU: нет
- NVMe SSD: 100 GB

**Плюсы**

- Генерацию забирает managed API, локальный GPU не нужен.
- Индекс и эмбеддинги живут у нас: по чувствительным документам можно ограничивать retrieval ещё до вызова LLM.
- Путь от текстов до рабочего прототипа короткий, а это в учебном спринте важнее всего остального.

**Минусы**

- Кусок контекста, который retrieval возвращает под конкретный запрос, всё-таки улетает в OpenAI.
- Появляются ключи, биллинг и отдельный вопрос: какие документы вообще допустимо отдавать в облако.

**Оценка**

Это мой основной вариант. По приватности он не идеален, но единственный, где мы за спринт успеваем довести пилот до конца без ручных подвигов.

### Вариант D. Гибридный / облачный с Yandex Cloud

**Стек**

- LLM: `YandexGPT Lite` или `YandexGPT Pro`.
- Embeddings: `all-MiniLM-L6-v2`.
- Vector DB: `FAISS` или `ChromaDB`, если позже понадобится richer metadata filtering.

**Рекомендуемый сервер**

- 8-16 vCPU
- 16-32 GB RAM
- GPU: нет
- NVMe SSD: 100 GB

**Плюсы**

- Managed-генерация без локального GPU, как и в варианте C.
- Токенизация подогнана под русский, поэтому на русскоязычных документах расход токенов будет заметно ниже OpenAI.
- Есть выбор между common instance, batch и dedicated — можно подстраиваться под нагрузку.

**Минусы**

- Для QuantumForge это второй облачный периметр поверх уже существующего AWS.
- Common instance делит ресурсы между клиентами, и latency под нагрузкой может плавать.
- Контекст уже, чем у `gpt-5-mini`: 32K против заметно большего окна у OpenAI.

**Оценка**

Вариант рабочий, но у C ничего не выигрывает. Имеет смысл только если проект уже живёт в Yandex Cloud или база почти вся на русском.

## 7. Итоговая рекомендация по результатам исследования

### Рекомендуемый стек для следующих заданий

- **LLM:** `gpt-5-mini`
- **Embeddings:** `sentence-transformers/all-MiniLM-L6-v2`
- **Векторная БД:** `FAISS`
- **Рекомендуемый сервер для пилота:** `8 vCPU / 16 GB RAM / 100 GB NVMe / без GPU`

### Почему именно такой стек

Инференс LLM отдаю в managed API, чтобы не поднимать GPU-сервер ради учебного пилота. Retrieval и эмбеддинги — локально: индекс под контролем, за эмбеддинги не платим, чувствительные документы при необходимости просто не попадают в outbound-запрос. FAISS — потому что на single-node и 30+ документах его хватает с запасом; фильтры и hybrid search Chroma пока некому загрузить.

### Что было рассмотрено и отложено

- Локальный LLM. Для 30+ документов и сроков спринта GPU-контур невыгоден. План Б на случай запрета cloud LLM.
- Облачные эмбеддинги. На таком объёме разница в качестве не окупает постоянный счёт и вынос документов во внешний API на этапе индексации.
- ChromaDB. Сильнее по фильтрам и метаданным, но на single-node пайплайне эти возможности сейчас нечем загрузить.
- YandexGPT. Рабочий, особенно на русском, но добавляет второй облачный периметр поверх AWS без явного выигрыша.

### Практическое правило по конфиденциальности

- Для synthetic KB из Задания 2 ограничений нет: можно использовать рекомендованный гибридный стек.
- Для реальной корпоративной документации сначала нужна классификация источников: `public / low sensitivity` можно отправлять в cloud LLM, а `confidential / regulated` лучше оставлять в local-only контуре.
